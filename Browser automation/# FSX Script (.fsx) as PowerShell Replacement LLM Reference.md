# F# Script (.fsx) as PowerShell Replacement: LLM Reference

## Context

The user operates on Windows and is migrating automation from PowerShell to F# scripts (`.fsx`, executed via `dotnet fsi`). The user knows PowerShell idioms well but prefers F# for type safety. When writing F# scripts, assume the user wants the operational outcome they would get from PowerShell, expressed in idiomatic F# using the .NET Base Class Library.

### Why F# Over PowerShell

PowerShell is dynamically typed, silently propagates `$null`, and coerces types implicitly; bugs surface only at runtime. When code is generated by an LLM, the target language's ability to **reject** incorrect code matters more than its ability to **express** correct code. A wrong guess in PowerShell silently runs; a wrong guess in F# fails to compile. Sparse F# training data plus a strict compiler outperforms abundant data plus no compiler.

Corollary: every pattern in this document is designed so that errors surface **at compile time or loudly at first run**, never silently.

---

## Execution Model

Single `.fsx` file, no project, no solution. Run via:

```
dotnet fsi script.fsx -- arg1 arg2
```

- Arguments: `fsi.CommandLineArgs` — index 0 is the script path; real arguments start at index 1.
- Script-relative paths: use the compile-time constant `__SOURCE_DIRECTORY__`, never the current working directory, for files that live next to the script.
- Exit codes: end every execution path with explicit `exit 0` / `exit 1`.

## Canonical Script Skeleton

Every generated script follows this shape:

```fsharp
// backup-logs.fsx — one-line statement of purpose
// Usage: dotnet fsi backup-logs.fsx -- <sourceDir> <destDir>

#r "nuget: CsvHelper"        // all #r directives first, before any open or code

open System
open System.IO

// ---- helpers ---------------------------------------------------------

// (run, requireSuccess, deserialize, etc. as needed — see patterns below)

// ---- main ------------------------------------------------------------

let main (argv: string[]) : int =
    eprintfn "[INFO] Starting with args: %A" argv
    // ... work, with diagnostics at every boundary ...
    eprintfn "[INFO] Done"
    0

let argv = fsi.CommandLineArgs |> Array.skip 1

try
    main argv |> exit
with ex ->
    eprintfn "[FATAL] %s" (ex.ToString())
    exit 1
```

The global `try/with` guarantees that any unhandled exception produces a full stack trace on stderr and a non-zero exit code — never a silent partial run.

## Script Modes: Contracted and Exploratory

Every script is one of two modes, decided up front from the situation. **Both modes are contracted** — they differ only in what the contract targets. A script is never contract-free; when the data cannot be typed yet, the _findings about the data_ are typed instead. The untyped region is quarantined to the smallest possible surface.

### Mode A — Contracted (strong)

When the domain is reasonably well understood — i.e. the case is not particularly convoluted and does not hinge on indeterminate external output — lay out the contract and types **early in the file**, immediately after the `open` declarations and before any logic:

```fsharp
open System
open System.IO

// ---- contract --------------------------------------------------------

type LogEntry = {
    Timestamp: DateTime
    Level: string
    Message: string
}

type Summary = {
    File: string
    Total: int
    Errors: int
}

// Function contracts, stated as signatures the implementations must satisfy:
//   parseEntry : string -> Result<LogEntry, string>
//   summarize  : string -> LogEntry[] -> Summary

// ---- helpers ---------------------------------------------------------
// ---- main ------------------------------------------------------------
```

This makes the type section function as a machine-checked specification: the compiler verifies every downstream function against the contract, and a reader (or the user reviewing LLM output) can validate the script's intent from the first screen without reading the implementation.

### Mode B — Exploratory (probe)

When the shape of the data is genuinely uncertain — external CLI tools with unknown output schemas, undocumented APIs, files of unknown structure — do **not** invent the data contract up front. That is exactly the guessing this document forbids. But the script is still contracted: **when you cannot type the data, type the findings.** The inspection itself gets a schema, even though the inspected data does not yet have one.

Instead of loose strings and dictionaries, the probe emits typed observation records:

```fsharp
// ---- contract (of the findings, not the data) --------------------------

type FieldObservation = {
    Path: string                 // JSON path / column name / line position
    ObservedKind: string         // "string" | "number" | "bool" | "array" | "object" | "null" | "mixed"
    SampleValues: string list    // up to N representative raw values
    PresentIn: int               // records containing this field
    TotalRecords: int
}

type ProbeReport = {
    Tool: string
    ArgsUsed: string list
    ExitCode: int
    RawSampleTruncated: string   // verbatim head of output, for human eyes
    Fields: FieldObservation list
}
```

The probe runs the tool, inspects the output, and prints the `ProbeReport` (serialized as JSON to stdout, traced on stderr). The user feeds the report back, and Phase 2 of _Discovery-First Typing_ derives the data contract from observed fields rather than guesses.

Applicability gradient: a trivial probe ("just show me what `netsh` prints") may dump raw output verbatim and nothing more. But the moment the probe **inspects** — counts fields, samples values, classifies kinds, aggregates anything — those inspections must flow through typed records, not ad-hoc printing. The rule of thumb: raw passthrough may be untyped; any computed observation may not.

The two modes are complementary, and one script may transition between them: an exploratory script, once its findings are confirmed, is rewritten (or extended) into a contracted script whose Mode A contract is copied from the Mode B findings.

## PowerShell-to-F# Mapping

PowerShell cmdlets are thin wrappers around .NET classes. Call the classes directly.

|PowerShell|F# (.NET)|
|---|---|
|`Get-ChildItem`|`Directory.EnumerateFiles`, `Directory.EnumerateDirectories`|
|`Get-Content`|`File.ReadAllText`, `File.ReadAllLines`, `File.ReadLines` (lazy)|
|`Set-Content` / `Out-File`|`File.WriteAllText`, `File.WriteAllLines`|
|`Copy-Item` / `Remove-Item`|`File.Copy`, `File.Delete`, `Directory.Delete`|
|`Test-Path`|`File.Exists`, `Directory.Exists`|
|`Get-Process` / `Start-Process`|`System.Diagnostics.Process`|
|`Invoke-WebRequest` / `Invoke-RestMethod`|`System.Net.Http.HttpClient`|
|`Get-Date`|`DateTime.Now`, `DateTimeOffset.Now`|
|`Select-String`|`System.Text.RegularExpressions.Regex`|
|`ConvertFrom-Json` / `ConvertTo-Json`|`System.Text.Json.JsonSerializer`|
|`Get-ItemProperty` (registry)|`Microsoft.Win32.Registry` (Windows; if types aren't found, add `#r "nuget: Microsoft.Win32.Registry"`)|
|`Get-CimInstance` / `Get-WmiObject`|`#r "nuget: System.Management"` → `ManagementObjectSearcher` (prefer this over shelling out to deprecated `wmic`)|

|`Measure-Object`|`Array.length`, `Array.sum`, `Array.average`|
|`Join-Path` / `Split-Path` / `Resolve-Path`|`Path.Combine` / `Path.GetDirectoryName`, `Path.GetFileName` / `Path.GetFullPath`|
|`Read-Host`|`Console.ReadLine()`|

### Pipeline → Functional Composition

```powershell
Get-ChildItem -Filter *.log | Where-Object { $_.Length -gt 1MB } | Sort-Object LastWriteTime
```

```fsharp
Directory.GetFiles(".", "*.log")
|> Array.map FileInfo
|> Array.filter (fun f -> f.Length > 1_048_576L)
|> Array.sortBy (fun f -> f.LastWriteTime)
```

### Null → Option

PowerShell silently propagates `$null`. F# makes absence explicit. Convert nullable .NET returns immediately at the boundary:

```fsharp
let tryGetEnv name =
    Environment.GetEnvironmentVariable(name) |> Option.ofObj
```

Then use `Option.map`, `Option.bind`, `Option.defaultValue` — never null checks downstream.

### Expected Failure → Result

Model expected failures (missing file, parse error) as `Result<'T, string>`. Reserve `try/with` for genuinely unexpected exceptions, and always trace via `eprintfn` before propagating:

```fsharp
let tryReadFile (path: string) : Result<string, string> =
    try File.ReadAllText path |> Ok
    with ex -> Error $"Failed to read {path}: {ex.Message}"
```

### String Interpolation

PowerShell `"$x"` / `"$(expr)"` → F# `$"text {x}"` / `$"text {expr}"`.

## Diagnostic Messaging (Mandatory)

Diagnostics go to **stderr** via `eprintfn` so they never contaminate stdout. This is the primary mechanism for verifying correctness on first run.

Place a trace at:

1. Every I/O boundary (file read/write, HTTP call, process invocation).
2. Every decision point — which branch of `if`/`match` was taken and why.
3. Every transformation that changes the count or shape of data — report before and after: `eprintfn "[FILTER] %d -> %d items after removing empty entries" before after`.
4. Every parse — report what succeeded and what was discarded.
5. Every chained dependency — trace the intermediate value being passed forward: `eprintfn "[CHAIN] Using account ID: %s" accountId`.

Prefixes: `[INFO]` `[STEP]` `[DATA]` `[FILTER]` `[PARSE]` `[CMD]` `[STDERR]` `[CHAIN]` `[PROBE]` `[TYPED]` `[FATAL]`.

## Culture Invariance (Critical on Non-US Windows)

`Double.Parse`, `DateTime.Parse`, and `.ToString()` honor the OS locale. On a machine with a comma decimal separator, `Double.Parse "3.14"` misbehaves. **All parsing and formatting of machine-generated text must pin the invariant culture:**

```fsharp
open System.Globalization
let inv = CultureInfo.InvariantCulture

let d  = Double.Parse("3.14", inv)
let dt = DateTime.ParseExact(s, "yyyy-MM-dd HH:mm:ss", inv)
let s2 = value.ToString("F2", inv)
```

Never call bare `Parse`/`ToString` on tool output or for output another tool will consume. (F#'s own `printf`/`sprintf` formatting is culture-invariant and safe.)

## External Tool Integration

### One Process Runner, Used Everywhere

Requirements: per-argument escaping (no hand-built quoted strings), both streams captured, no pipe deadlock, exit code always traced.

```fsharp
open System.Diagnostics

type RunResult = {
    Command: string
    ExitCode: int
    Stdout: string
    Stderr: string
}

/// Args are passed as a list; ArgumentList escapes each one correctly.
let run (cmd: string) (args: string list) : RunResult =
    let psi =
        ProcessStartInfo(
            cmd,
            RedirectStandardOutput = true,
            RedirectStandardError = true,
            UseShellExecute = false)
    args |> List.iter psi.ArgumentList.Add
    use proc = Process.Start psi
    // Read stderr asynchronously while stdout is read synchronously.
    // Sequential ReadToEnd() on both streams DEADLOCKS when the child
    // fills one pipe buffer while we block on the other.
    let stderrTask = proc.StandardError.ReadToEndAsync()
    let stdout = proc.StandardOutput.ReadToEnd()
    proc.WaitForExit()
    let stderr = stderrTask.Result
    let display = cmd + " " + String.concat " " args
    eprintfn "[CMD] %s -> exit %d" display proc.ExitCode
    if stderr.Length > 0 then eprintfn "[STDERR] %s" (stderr.TrimEnd())
    { Command = display; ExitCode = proc.ExitCode; Stdout = stdout; Stderr = stderr }

let requireSuccess (r: RunResult) : string =
    if r.ExitCode <> 0 then
        failwithf "Command failed (exit %d): %s\n%s" r.ExitCode r.Command r.Stderr
    r.Stdout
```

For tools that can hang, use `proc.WaitForExit(timeoutMs)` and on timeout `proc.Kill(entireProcessTree = true)` before failing.

A zero exit code with empty stdout is not the same as a non-zero exit code — trace both, assess both.

### Discovery-First Typing: Probe → Type → Parse

The LLM does not reliably know the exact output format of every CLI tool and version. Guessing the schema reproduces the silent-failure class this migration exists to eliminate.

**Phase 1 — Probe.** Run the tool with safe, read-only parameters. Prefer JSON modes (`--output json`, `-o json`, `--format json`). Log output verbatim:

```fsharp
let probe = run "az" ["account"; "list"; "--output"; "json"] |> requireSuccess
eprintfn "[PROBE] az account list returned:\n%s" probe
```

If the probe does any inspection beyond raw passthrough — sampling fields, counting occurrences, classifying value kinds — structure those observations as a typed `ProbeReport` (see _Mode B — Exploratory_), so the inspection itself has a compiler-checked contract even though the data does not yet.

When uncertain about a tool's output, generate the probe as a standalone script and stop:

```
// TODO: Run this script and provide the output.
// I will define typed records from the actual output shape
// rather than guessing the schema.
```

**Phase 2 — Type.** Define records matching the observed output. Use `JsonPropertyName` for foreign naming conventions; use `option` for sometimes-absent fields; type numerics and booleans precisely — never default to `string`:

```fsharp
open System.Text.Json.Serialization

type AzAccount = {
    [<JsonPropertyName("id")>] Id: string
    [<JsonPropertyName("name")>] Name: string
    [<JsonPropertyName("isDefault")>] IsDefault: bool
    [<JsonPropertyName("state")>] State: string
}
```

For non-JSON output, define a record plus a line parser returning `Result<'T, string>`, with column positions or delimiters documented explicitly in comments.

**Phase 3 — Parse and Validate.** Deserialize, then trace what parsed and what didn't:

```fsharp
let accounts = deserialize<AzAccount[]> probe
eprintfn "[TYPED] Parsed %d accounts" accounts.Length

// Non-JSON path:
let parsed = lines |> Array.map parseLine
let oks  = parsed |> Array.choose (function Ok v -> Some v | _ -> None)
let errs = parsed |> Array.choose (function Error e -> Some e | _ -> None)
eprintfn "[PARSE] %d parsed, %d failed" oks.Length errs.Length
errs |> Array.iter (eprintfn "[PARSE:FAIL] %s")
```

## JSON

```fsharp
open System.Text.Json

let jsonOptions = JsonSerializerOptions(WriteIndented = true, PropertyNameCaseInsensitive = true)

let serialize (value: 'T) : string = JsonSerializer.Serialize(value, jsonOptions)
let deserialize<'T> (json: string) : 'T = JsonSerializer.Deserialize<'T>(json, jsonOptions)
```

On .NET 6+, `System.Text.Json` natively handles F# records, `option`, lists, sets, and maps. It does **not** handle discriminated unions — if a DU must round-trip through JSON, add:

```fsharp
#r "nuget: FSharp.SystemTextJson"
// then: jsonOptions.Converters.Add(JsonFSharpConverter())
```

## Common Patterns

### HTTP — One Shared Client

```fsharp
open System.Net.Http

let http = new HttpClient()   // ONE instance per script; per-call instances exhaust sockets in loops

let httpGet (url: string) : string =
    let body = http.GetStringAsync(url).Result   // .Result is safe in fsi (no sync context)
    eprintfn "[HTTP] GET %s -> %d chars" url body.Length
    body
```

### Recursive Directory Traversal

```fsharp
let allFiles (dir: string) (pattern: string) : string seq =
    let opts = EnumerationOptions(RecurseSubdirectories = true, IgnoreInaccessible = true)
    Directory.EnumerateFiles(dir, pattern, opts)
```

`IgnoreInaccessible` prevents a single permission-denied subdirectory from aborting the whole traversal.

### CSV

Naive splitting is acceptable **only** for trusted, simple data — it breaks silently on quoted fields containing commas:

```fsharp
// WARNING: no quoted-field support. For real-world CSV use CsvHelper
// (#r "nuget: CsvHelper") or FSharp.Data's CsvFile.
let readCsvNaive (path: string) : string[][] =
    File.ReadAllLines path |> Array.map (fun line -> line.Split ',')
```

If the input's provenance is unknown, default to the library, not the split.

### Temporary Files

```fsharp
let withTempFile (extension: string) (action: string -> 'T) : 'T =
    let path = Path.Combine(Path.GetTempPath(), Guid.NewGuid().ToString("N") + extension)
    eprintfn "[TEMP] Created %s" path
    try action path
    finally
        if File.Exists path then
            File.Delete path
            eprintfn "[TEMP] Deleted %s" path
```

### Console Encoding (Windows)

If output contains non-ASCII text, set once at script start:

```fsharp
Console.OutputEncoding <- Text.Encoding.UTF8
```

For external tools that emit UTF-8 on a machine with a legacy console codepage, also set `psi.StandardOutputEncoding <- Text.Encoding.UTF8` in the runner.

## Rules

1. All `#r` directives first, then `open` declarations. No fully qualified names inline except for disambiguation.
2. Prefer immutable `let`. Use `mutable` only for .NET interop that requires it.
3. Prefer `Seq` for lazy/large data, `Array` for random access and performance, `List` for recursive pattern matching.
4. Never silently discard errors: nullable returns → `Option.ofObj` immediately; exceptions → `try/with` plus an `eprintfn` trace.
5. Wrap `main` in a global `try/with` that prints `[FATAL]` with the full exception and exits 1. End every path with explicit `exit 0` / `exit 1`.
6. Exactly one process runner (`run` above): per-argument `ArgumentList`, both streams captured separately, deadlock-safe, exit code traced and checked on every invocation.
7. Never guess external tool output schema. If uncertain, emit a probe-only script and request actual output before writing the parser.
8. Prefer `--output json` flags wherever a tool supports them; structured parsing beats regex on human-readable text. For tabular text, document delimiters/columns in comments and trace sample parsed rows.
9. Pin `CultureInfo.InvariantCulture` on every numeric/date parse and every formatted output destined for another tool.
10. Annotate types at all system boundaries; when uncertain about an API signature, state the uncertainty in a comment rather than guessing.
11. Diagnostics per the Diagnostic Messaging section at every I/O boundary, decision point, and shape-changing transformation. Not optional.
12. Decide the script mode up front. **Contracted mode** (well-understood case): lay out the data contract and types early in the file, immediately after `open` declarations. **Exploratory mode** (indeterminate data shape): probe first (Rule 7) and derive the data contract from observed output.
13. Exploratory scripts are still contracted: when the data cannot be typed, type the findings. Raw passthrough of output may be untyped; any computed observation (field samples, counts, kind classification) must flow through typed records such as `ProbeReport`. No script is contract-free.
